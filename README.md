Firebase handles authentication and database, Cloudinary handles image storage, and Vercel handles deployment and serverless functions.

## Project Overview

WanderLustBagsPH is an online shop offering affordable travel bags, accessories, and travel essentials. The application supports two types of users:

- **Customers** — can browse products, manage a cart, place orders, and manage their profile and delivery addresses.
- **Admins** — can manage products (CRUD with image uploads), view and update orders, manage user accounts, and configure payment methods through a dedicated admin panel.

## Features

### Customer-Facing
- User registration, login, logout, and password reset
- Browse all products with search and category filtering
- Product detail pages with quantity selection and stock display
- Add to cart, update quantities, remove items
- Checkout with proof of payment upload (images only) and delivery address selection
- Order tracking page with real-time status updates, search by order ID / customer / date / payment method, proof of payment preview, and delivery tracking info
- Order summary shows items total, cash-on-delivery shipping fee (if set by admin), and total due
- Profile management: edit name and contact, upload avatar via drag-and-drop (Cloudinary)
- Multiple delivery addresses with default address support
- Account deletion with re-authentication

### Admin Panel
- Separate admin login with `isAdmin` Firestore flag check
- Dashboard with stats: total revenue, products, orders, users; recent orders table; low-stock alerts
- Full product CRUD: add, edit, delete with drag-and-drop Cloudinary image upload
- Order management: view all orders, update order statuses, cancel orders (with automatic stock restoration), view full order details, add delivery tracking info, and set cash-on-delivery shipping fee per order
- User management: view all registered users and their roles
- Admin profile editing
- **Payment Methods management**: Add, edit, delete custom payment methods (e-wallets, bank transfers); Active/Inactive toggle controls visibility on checkout

### 2. Firebase Setup

1. Go to [console.firebase.google.com](https://console.firebase.google.com) and create a new project (or use an existing one).

2. **Enable Authentication:**
   - In the Firebase console, go to **Build → Authentication → Sign-in method**
   - Enable **Email/Password**

3. **Create a Firestore database:**
   - Go to **Build → Firestore Database → Create database**
   - Choose your preferred region (the project uses `asia-southeast1`)
   - Start in **test mode** for development (apply proper security rules before going to production)

4. **Get your Firebase config:**
   - Go to **Project Settings → General → Your apps**
   - Click **Add app → Web**, register your app
   - Copy the `firebaseConfig` object — you'll need each field for the `.env` file

5. **Apply Firestore security rules** (optional but recommended for production):

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function isSignedIn() {
      return request.auth != null;
    }

    function isAdmin() {
      return isSignedIn() &&
        get(/databases/$(database)/documents/users/$(request.auth.uid)).data.isAdmin == true;
    }

    function isSelf(userId) {
      return isSignedIn() && request.auth.uid == userId;
    }

    match /products/{productId} {
      allow read: if true;
      allow create, update, delete: if isAdmin();
    }

    match /orders/{orderId} {
      allow create: if isSignedIn() && request.resource.data.customerId == request.auth.uid;
      allow read, update, delete: if isAdmin() || resource.data.customerId == request.auth.uid;
    }

    match /users/{userId} {
      allow create: if isSignedIn() && request.auth.uid == userId;
      allow read, update, delete: if isSelf(userId) || isAdmin();

      match /addresses/{addressId} {
        allow read, write, delete: if isSelf(userId);
      }
    }

    match /carts/{userId}/items/{itemId} {
      allow read, write, delete: if isSignedIn() && request.auth.uid == userId;
    }
  }
}
```

### 7. Creating an Admin Account

Admin accounts are not created through the UI. To make a user an admin:

1. Register a new account normally through the app's registration page.
2. Go to your **Firebase Console → Firestore Database → users collection**.
3. Find the document for the user you want to promote.
4. Add or edit the field: `isAdmin` → `true` (boolean).

That user can now log in through `admin_login.html` and access the admin panel.

> The admin login page verifies the `isAdmin` flag in Firestore. Regular users who attempt to log in via the admin login page will be denied access even with valid credentials.

## Important Notes

- **Firebase CDN imports only** — this project does not use npm-installed Firebase packages on the frontend. All Firebase SDK imports use CDN URLs (`https://www.gstatic.com/firebasejs/10.12.0/...`). Do not use `import` statements inside inline `<script>` tags in HTML — they must go in external `.js` files with `type="module"`.
- **No Firebase Storage** — image uploads go through Cloudinary (unsigned preset) because Firebase Storage requires a paid Blaze plan. Secure deletion of Cloudinary images is handled by a Vercel serverless function.
- **Passwords are never stored** — Firebase Authentication handles all credential management. No passwords are written to Firestore.
- **`firebase-config.js` is gitignored** — it contains secrets and must be generated locally via `npm run build` or by Vercel during deployment.
- **Admin detection is Firestore-based** — the `isAdmin: true` field on a user's Firestore document is the single source of truth for admin access.

## Key File Contents (for AI Agent Context)

This section documents the exact contents of the three most critical build/config files. AI agents and new contributors can use this as a reference when the actual files are not available (e.g., `firebase-config.js` being gitignored).

### `build.js` — Config Generator Script

This Node.js script reads the `.env` file in the project root and substitutes the values into `firebase-config-template.js`, writing the result to `firebase-config.js`. It is run automatically by Vercel during deployment via `npm run build`.

```js
const fs = require('fs');
const path = require('path');

function parseEnv(filePath) {
  const env = {};
  if (!fs.existsSync(filePath)) return env;
  const content = fs.readFileSync(filePath, 'utf8');
  content.split(/\r?\n/).forEach(line => {
    line = line.trim();
    if (!line || line.startsWith('#')) return;
    const [key, ...rest] = line.split('=');
    if (key) env[key.trim()] = rest.join('=').trim().replace(/^"|"$/g, '');
  });
  return env;
}

const envPath = path.join(__dirname, '.env');
const env = parseEnv(envPath);

const configPath = path.join(__dirname, 'wanderlust_main', 'js', 'firebase-config-template.js');
const outputPath = path.join(__dirname, 'wanderlust_main', 'js', 'firebase-config.js');

if (!fs.existsSync(configPath)) {
  console.error('Missing firebase-config-template.js');
  process.exit(1);
}

let template = fs.readFileSync(configPath, 'utf8');

template = template.replace('{{VITE_FIREBASE_API_KEY}}', env.VITE_FIREBASE_API_KEY || '');
template = template.replace('{{VITE_FIREBASE_AUTH_DOMAIN}}', env.VITE_FIREBASE_AUTH_DOMAIN || '');
template = template.replace('{{VITE_FIREBASE_PROJECT_ID}}', env.VITE_FIREBASE_PROJECT_ID || '');
template = template.replace('{{VITE_FIREBASE_STORAGE_BUCKET}}', env.VITE_FIREBASE_STORAGE_BUCKET || '');
template = template.replace('{{VITE_FIREBASE_MESSAGING_SENDER_ID}}', env.VITE_FIREBASE_MESSAGING_SENDER_ID || '');
template = template.replace('{{VITE_FIREBASE_APP_ID}}', env.VITE_FIREBASE_APP_ID || '');
template = template.replace('{{VITE_FIREBASE_MEASUREMENT_ID}}', env.VITE_FIREBASE_MEASUREMENT_ID || '');

fs.writeFileSync(outputPath, template);
console.log('firebase-config.js generated successfully');
```

### `api/delete-cloudinary.js` — Serverless Image Deletion

This Vercel Serverless Function handles secure deletion of images from Cloudinary. It is called from the frontend after a product or user avatar is deleted. It requires `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, and `CLOUDINARY_API_SECRET` to be set as environment variables in Vercel.

```js
const cloudinary = require('cloudinary').v2;

cloudinary.config({
    cloud_name: process.env.CLOUDINARY_CLOUD_NAME,
    api_key: process.env.CLOUDINARY_API_KEY,
    api_secret: process.env.CLOUDINARY_API_SECRET
});

module.exports = async (req, res) => {
    if (req.method !== 'POST') {
        return res.status(405).json({ error: 'Method Not Allowed' });
    }

    try {
        const { public_id } = req.body;

        if (!public_id) {
            return res.status(400).json({ error: 'Missing public_id' });
        }

        const result = await cloudinary.uploader.destroy(public_id);

        res.status(200).json({ result });
    } catch (error) {
        res.status(500).json({ error: error.message });
    }
};
```

## AI Agent Guidelines

If you are an AI agent working on this codebase, keep the following constraints in mind at all times:

- **No npm/Node.js on the frontend.** All Firebase imports must use CDN URLs (`https://www.gstatic.com/firebasejs/10.12.0/...`). Never suggest installing Firebase via npm for frontend files.
- **No inline module imports.** `import` statements cannot be placed inside `<script>` blocks directly in HTML. They must be in external `.js` files loaded with `<script type="module" src="...">`.
- **No Firebase Storage.** All image uploads and deletions go through Cloudinary. Upload uses the unsigned preset `WanderLustBagsPH` on cloud `dupigtgx6`. Deletion goes through the Vercel function at `/api/delete-cloudinary`.
- **`firebase-config.js` does not exist in the repo.** It is generated by `build.js`. If any JS file needs Firebase, it imports from `./firebase-config.js` (or a relative equivalent). Never hardcode credentials.
- **Admin security is handled by `admin_security.js`.** All admin pages call `onAdminStateChanged()` from this module, which checks `isAdmin === true` in the user's Firestore document. Never rely on email whitelists or session storage alone for access control.
- **Firestore is the database.** There is no MySQL, no REST API, no backend server. All data reads and writes go directly to Firestore using the CDN SDK.
- **Filename consistency is critical.** JS files use `snake_case` (e.g., `admin_products.js`). Mismatched filenames cause silent 404 failures with no error in the console.
- **The project ID is `wanderlustbagsph-816c1`.** Firestore is hosted in `asia-southeast1`.
- **Working branch is `Payment-Test`.** Use GitHub Desktop for version control.
- **Cloudinary config:** Cloud name `dupigtgx6`, upload preset `WanderLustBagsPH`, unsigned mode. Store both `image` (URL) and `publicId` (or `avatarPublicId` for users) in Firestore so images can be deleted later.
- **Design system:** Font is `Playfair Display` (Google Fonts). Brown palette: `#5c491f` (dark), `#7c6c3f` (mid), `#797259` (muted), `#fffef8` (off-white background), `#e2d7c3` (light beige). All buttons use `border-radius: 5px`. Admin pages use `admin_style.css`. Customer pages use their own per-page stylesheets.
- **Firestore Collections:** `products`, `orders`, `users` (with subcollection `addresses`), `carts/{userId}/items`, `paymentMethods`.
